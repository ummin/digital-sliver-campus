# 디지털 실버 캠퍼스 개발 로드맵

중장년·노년층이 키오스크·스마트폰 앱·AI 도구 사용법을 가장 쉬운 단계부터 따라 배우는 학습 플랫폼

## 개요

디지털 실버 캠퍼스는 디지털 기기에 익숙하지 않은 50대 이상 중장년·노년층 학습자를 위한 비로그인 실습 학습 서비스로 다음 기능을 제공합니다:

- **카테고리 탐색**: 키오스크 / 스마트폰 앱 / AI 도구 학습 주제를 큰 카드로 노출
- **난이도순 실습 목록**: 가장 쉬운 단계부터 따라갈 수 있도록 정렬된 콘텐츠 목록
- **단계별 실습 뷰어**: 한 화면에 한 단계씩 이미지·영상과 함께 안내하는 학습 화면
- **피드백/문의 제출**: 학습 중 막히는 부분을 운영자에게 전달하는 문의 채널
- **접근성 모드**: 글씨 크게/고대비 토글로 노년층 가독성 보장

## 개발 워크플로우

1. **작업 계획**

- 기존 코드베이스를 학습하고 현재 상태를 파악
- 새로운 작업을 포함하도록 `ROADMAP.md` 업데이트
- 우선순위 작업은 마지막 완료된 작업 다음에 삽입

2. **작업 생성**

- 기존 코드베이스를 학습하고 현재 상태를 파악
- `/tasks` 디렉토리에 새 작업 파일 생성
- 명명 형식: `XXX-description.md` (예: `001-setup.md`)
- 고수준 명세서, 관련 파일, 수락 기준, 구현 단계 포함
- **API/비즈니스 로직 작업 시 "## 테스트 체크리스트" 섹션 필수 포함 (Playwright MCP 테스트 시나리오 작성)**
- 예시를 위해 `/tasks` 디렉토리의 마지막 완료된 작업 참조

3. **작업 구현**

- 작업 파일의 명세서를 따름
- 기능과 기능성 구현
- **API 연동 및 비즈니스 로직 구현 시 Playwright MCP로 테스트 수행 필수**
- 각 단계 후 작업 파일 내 단계 진행 상황 업데이트
- 구현 완료 후 Playwright MCP를 사용한 E2E 테스트 실행
- 테스트 통과 확인 후 다음 단계로 진행
- 각 단계 완료 후 중단하고 추가 지시를 기다림

4. **로드맵 업데이트**

- 로드맵에서 완료된 작업을 ✅로 표시

## 개발 단계

### Phase 1: 애플리케이션 골격 구축

- **Task 001: 라우트 구조 및 빈 페이지 정리** - 우선순위
  - PRD 메뉴 구조(홈/배우기/실습 목록/문의하기)에 맞춰 라우트 정리: `app/page.tsx`(홈), `app/learn/page.tsx`(카테고리 목록), `app/contents/page.tsx`(실습 콘텐츠 목록), `app/contact/page.tsx`(피드백 폼)
  - 실습 콘텐츠 뷰어 동적 라우트 생성: `app/contents/[id]/page.tsx`
  - 카테고리별 실습 목록 라우트 생성: `app/learn/[categoryId]/page.tsx` (또는 쿼리 파라미터로 `contents` 필터링 방식 결정)
  - 각 페이지 빈 껍데기(타이틀 + TODO 주석) 작성, 기존 `app/components/page.tsx`(컴포넌트 갤러리)는 개발용으로 유지
  - `SiteHeader`/`SiteFooter`의 NAV 상수에 배우기/실습 목록/문의하기 메뉴 반영

- **Task 002: 타입 정의 및 Notion 스키마 설계**
  - `lib/types.ts`(또는 `types/` 디렉토리)에 카테고리·실습 콘텐츠·피드백 TypeScript 인터페이스 정의 (PRD 데이터 모델 기준)
  - Notion 데이터베이스 스키마 문서화: 카테고리 DB(제목/설명/아이콘/정렬순서), 실습 콘텐츠 DB(제목/카테고리 Relation/난이도/순서/본문), 피드백 DB(이름/연락처/문의내용/첨부/관련콘텐츠/상태)
  - API 응답/요청 타입 정의 (Notion 조회 결과 매핑 타입, 피드백 제출 요청 타입)
  - 환경변수 명세 정리: `NOTION_TOKEN`, `NOTION_CATEGORY_DB_ID`, `NOTION_CONTENT_DB_ID`, `NOTION_FEEDBACK_DB_ID`, `BLOB_READ_WRITE_TOKEN` 등 `.env.example` 작성

### Phase 2: UI/UX 완성 (더미 데이터 활용)

- **Task 003: 접근성 모드(글씨 크게/고대비) 구현** - 우선순위
  - F011: 헤더 공통 영역에 글씨 크게 토글 버튼 추가 (Zustand store로 전역 상태 관리)
  - 고대비 모드 CSS 변수(`app/globals.css` `:root`/`.dark` 블록 확장 또는 별도 `data-attribute` 기반 테마) 정의
  - 토글 상태 `localStorage` 영속화 (새로고침 시 유지)
  - 모든 페이지에서 폰트 크기 확대 시 레이아웃 깨짐 없는지 반응형 검증

- **Task 004: 홈 페이지 UI 완성 (더미 데이터)**
  - F001/F010: 큰 글씨 환영 문구 + 서비스 소개 섹션
  - 카테고리 대표 카드 3종(키오스크/스마트폰 앱/AI 도구) 더미 데이터로 노출
  - "배우기 시작하기" CTA 버튼 → 카테고리 목록 페이지 연결
  - 노년층 친화 큰 아이콘·큰 글씨·고대비 색상 적용

- **Task 005: 카테고리 목록 페이지 UI 완성 (더미 데이터)**
  - F001: 더미 카테고리 데이터를 카드 그리드로 노출 (아이콘·제목·간단 설명)
  - 카드 클릭 시 실습 콘텐츠 목록 페이지로 이동하는 네비게이션 연결
  - 반응형 그리드(모바일 1열/태블릿 2열/데스크탑 3열)

- **Task 006: 실습 콘텐츠 목록 페이지 UI 완성 (더미 데이터)**
  - F002: 더미 실습 콘텐츠를 난이도/순서 필드 기준 오름차순 정렬해 카드로 노출
  - 카드에 제목·난이도 배지("초보자 추천" 등)·대표 이미지 표시
  - 카드 클릭 시 실습 콘텐츠 뷰어 페이지로 이동

- **Task 007: 실습 콘텐츠 뷰어 페이지 UI 완성 (더미 데이터)**
  - F003: 더미 단계(step) 데이터를 한 화면에 한 단계씩 렌더링 (설명 + 이미지/영상)
  - 이전/다음 단계 이동 버튼 + 진행 표시(예: "3/7 단계") 구현
  - 마지막 단계 완료 시 "궁금한 점 있음"(→ 피드백 폼 진입) / "다른 주제 학습"(→ 카테고리 목록 복귀) 분기 UI
  - 큰 이미지/영상 첨부 영역 반응형 처리

- **Task 008: 피드백 폼 페이지 UI 완성 (더미 제출)**
  - F004: 이름/익명 선택 토글, 연락처(선택)·문의내용(필수) 입력 폼 마크업
  - React Hook Form + Zod로 클라이언트 측 유효성 검증 (익명 선택 시 연락처 선택적 처리하는 조건부 스키마)
  - 파일 첨부 UI(이미지 미리보기 포함, 실제 업로드는 더미 처리)
  - 제출하기 버튼 클릭 시 더미 성공 토스트(sonner) 노출

### Phase 3: 핵심 기능 구현

- **Task 009: Notion API 연동 - 카테고리/실습 콘텐츠 조회** - 우선순위
  - `@notionhq/client` 패키지 설치 및 `lib/notion.ts` 클라이언트 초기화
  - 카테고리 DB 조회 함수 구현 (정렬순서 오름차순) → 홈/카테고리 목록 페이지 더미 데이터 교체
  - 실습 콘텐츠 DB 조회 함수 구현 (카테고리 Relation 필터 + 난이도/순서 오름차순) → 실습 콘텐츠 목록 페이지 더미 데이터 교체
  - 실습 콘텐츠 본문(Notion 블록) 조회 및 단계(step)별 파싱 로직 구현 → 실습 콘텐츠 뷰어 페이지 더미 데이터 교체
  - Notion 호스팅 파일 URL 1시간 만료 대응: 콘텐츠 페이지 `revalidate` 3600초 미만 설정 또는 Vercel Blob external URL 참조 확인
  - Playwright MCP로 홈 → 카테고리 → 실습 목록 → 뷰어까지 실제 데이터 노출 E2E 테스트

- **Task 010: Vercel Blob 연동 - 피드백 첨부 파일 업로드**
  - `@vercel/blob` 패키지 설치 및 업로드 API 라우트 구현 (`app/api/upload/route.ts`)
  - 피드백 폼에서 파일 선택 시 Vercel Blob에 업로드 후 영구 URL 반환받는 로직 연결
  - Zod로 파일 크기·확장자(이미지 위주) 검증 강화
  - Playwright MCP로 파일 첨부 업로드 플로우 테스트

- **Task 011: 피드백/문의 제출 API 구현**
  - F004: 피드백 제출 API 라우트 구현 (`app/api/feedback/route.ts`) — Notion 피드백 DB에 저장
  - 익명 제출 분기 처리: "익명" 선택 시 이름 고정 + 연락처 선택적 검증 (Zod 조건부 스키마)
  - 관련콘텐츠 Relation 연결 (실습 콘텐츠 뷰어에서 진입 시 콘텐츠 ID 전달)
  - 제출 성공/실패 UI 분기 (성공 시 안내 후 홈 또는 직전 콘텐츠로 이동, 실패 시 에러 표시)
  - Playwright MCP로 피드백 제출 전체 플로우(익명/실명, 파일 첨부 유무) E2E 테스트

- **Task 011-1: 핵심 기능 통합 테스트**
  - Playwright MCP를 사용한 전체 사용자 여정 테스트 (홈 → 카테고리 → 실습 목록 → 뷰어 → 피드백 제출)
  - Notion API 응답 지연/오류 시 에러 핸들링 검증
  - 접근성 모드(글씨 크게/고대비) 켠 상태에서 전체 플로우 회귀 테스트
  - 빈 카테고리/빈 콘텐츠 목록 등 엣지 케이스 UI 검증

  - 이미지/영상 로딩 실패 시 대체 UI(placeholder) 처리

  - 이미지 최적화(`next/image`) 적용 및 Vercel Blob URL과의 연동 확인
  - Lighthouse 접근성/성능 점수 측정 및 개선 (노년층 대상 최소 로딩 시간 목표)

- **Task 014: 배포 파이프라인 구축**
  - Vercel 프로젝트 연결 및 환경변수(Notion 토큰, Blob 토큰) 설정
  - 프로덕션 빌드 검증 (`npm run build`) 및 배포 후 스모크 테스트
  - 운영자용 Notion 워크스페이스 접근 가이드 문서화 (콘텐츠/피드백 관리 방법)
